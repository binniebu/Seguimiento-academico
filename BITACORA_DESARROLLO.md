# Bitacora de Desarrollo - Sistema de Seguimiento Academico

Este documento sirve para registrar de manera clara y descriptiva todos los cambios, refactorizaciones y preparativos realizados en el codigo fuente y la base de datos del sistema durante la fase de desarrollo.

---

## Historial de Cambios

### 01/07/2026 - Preparacion de Arquitectura y Base de Datos (Completado)
* **Estado:** Modificaciones de base de datos e infraestructura basica finalizadas con exito.
* **Cambios realizados:**
  * **Base de Datos:** Se crearon las tablas de carreras (carreras), periodos academicos (periodos_academicos), secciones (secciones) y la tabla intermedia de permisos multiples (usuarios_roles).
  * **Base de Datos:** Se realizo la migracion de los roles de usuario existentes hacia la nueva tabla intermedia para mantener la compatibilidad con las cuentas actuales.
  * **Base de Datos:** Se agregaron las columnas de creditos a las materias, la ruta del expediente en formato PDF a los usuarios (con estado pendiente por defecto para el registro publico) y el campo de estado de ciclo de vida a los estudiantes.
  * **Base de Datos:** Se ejecuto el script completo de actualizacion en el servidor local de MySQL.
  * **Enrutamiento:** Se registraron en el archivo index.php los accesos y controladores para carreras, secciones, solicitudes de pre-registro, historial academico y la funcion switch_role.
  * **Middleware:** Se modifico el archivo RoleMiddleware.php para registrar los accesos de cada rol y se añadio una validacion para cerrar la sesion de estudiantes que se encuentren bloqueados o inactivos.
  * **Capa de Modelos (DAO):** Se crearon los archivos CarreraDao.php, SeccionDao.php, SolicitudDao.php e HistorialDao.php para centralizar la logica de base de datos de los nuevos modulos.
  * **Capa de Modelos (DAO):** Se modificaron las funciones eliminarEstudiante (en EstudianteDao.php) y eliminarMaestro (en MaestroDao.php) para reemplazar las sentencias SQL DELETE por UPDATE. Ahora el sistema realiza un borrado logico inactivando los estados de las cuentas en lugar de destruirlas fisicamente en la base de datos, protegiendo asi la integridad historica de las matriculas y calificaciones.
  * **Capa de Modelos (DAO):** Se actualizo la funcion obtenerEstudiantes (en EstudianteDao.php) para filtrar de forma automatica y excluir de la lista a aquellos estudiantes cuyo estado sea inactivo, logrando que desaparezcan de la bandeja al ser dados de baja. Asimismo, se implemento la busqueda directa por ID numerico exacto de estudiante.
  * **Conmutador de Roles (Role Switcher):** Se implemento un control desplegable de seleccion de rol en el Home del sistema que permite a usuarios con multiples roles asignados cambiar de vista de forma instantanea sin cerrar sesion.
  * **Interfaz de Usuario (Estudiantes):** Se rediseño la vista estudiantes.view.tpl para soportar la alternancia entre la lista de alumnos activos y la nueva bandeja de "Bajas / Reingresos" (empleando el parametro ver=inactivos), adaptando los titulos, botones de accion (como "Activar Cuenta") y los colores de los badges de estado.
  * **Interfaz de Usuario (Estudiantes):** Se rediseño el boton de la tabla de estudiantes, renombrando la accion de "Eliminar" a "Dar de baja" (y actualizando sus iconos y textos de confirmacion) para reflejar con exactitud la operacion de borrado logico implementada.
  * **Interfaz de Usuario (Estudiantes):** Se modifico el formulario de registro y edicion estudiante_nuevo.view.tpl reemplazando el campo de texto libre de Carrera por una entrada de texto inteligente vinculada a un elemento datalist dinamico alimentado por la tabla de carreras. Esto proporciona un buscador predictivo mientras el usuario escribe, y se agrego una validacion Javascript al enviar el formulario para garantizar que la carrera seleccionada exista oficialmente en la institucion. Asimismo, se añadio una funcion helper fixDoubleEncoding para sanitizar caracteres con acentos mal codificados provenientes de la base de datos (evitando simbolos corruptos como ├│) mediante el uso de mb_convert_encoding para garantizar compatibilidad con PHP 8.2 y omitir advertencias de deprecacion, se removio el campo de contraseñas al editar un registro para evitar confusiones de seguridad y se bloqueo el campo de Carrera como solo lectura (inhabilitado) en el modo edicion para impedir que el administrador altere la carrera cursada de manera arbitraria.
  * **Interfaz de Usuario (Home):** Se rediseño por completo la pagina home.view.tpl reemplazando el menu de navegacion superior por una barra lateral izquierda (Sidebar) uniforme con el resto del sistema, garantizando la consistencia visual y de navegacion.
  * **Interfaz de Usuario (Sidebar):** Se añadio un boton de cierre (x) dentro de la barra lateral al lado del titulo MENU y un boton de menu (hamburguesa) en la cabecera principal. El boton de la cabecera se oculta de forma automatica mediante CSS cuando la barra lateral esta abierta y se visualiza unicamente cuando esta colapsada, logrando una interfaz limpia y fluida para el usuario. Se homogeneizo ademas la barra lateral en todas las vistas del sistema (estudiantes, carreras, etc.) para que se construya de manera dinamica evaluando el rol del usuario en la sesion actual.
  * **Interfaz de Usuario (Paginación y Filtros):** Se incorporo un control de paginacion dinamico de Bootstrap al pie de la tabla de estudiantes, mostrando 5 registros por pagina. Asimismo, se agrego un boton rojo de "Limpiar" en el formulario de busqueda que se activa cuando existe un filtro activo, permitiendo restablecer la lista completa de alumnos con un solo clic. Se adecuaron ademas las alertas de bandeja vacia para que muestren mensajes contextuales sin instigar a registrar nuevos estudiantes cuando se esta en el panel de bajas o si no existen coincidencias de busqueda.
  * **Interfaz de Usuario (Estilos):** Se agrego al archivo public/css/style.css un sistema de diseño premium, definiendo tipografia Inter, gradientes oscuros para la barra lateral (sidebar), bordes suavizados, tarjetas contenedoras sombreadas, badges redondeados y micro-transacciones visuales en botones y enlaces. Ademas, se redujo el grosor de los botones (ajustando el padding de btn y btn-sm) para lograr botones mas estilizados y elegantes, y se estructuraron las acciones de la tabla en dos filas verticales mediante una caja flex-column con la regla white-space: nowrap para evitar el recorte de informacion y la necesidad de barras de navegacion en pantallas grandes.
  * **Capa de Modelos (DAO):** Se modifico EstudianteDao.php para admitir parametros de limit y offset en obtenerEstudiantes y se añadio la funcion obtenerTotalEstudiantes para retornar la sumatoria de coincidencias, habilitando asi la paginacion dinamica a nivel de base de datos.
  * **Capa de Modelos (DAO):** Se rediseño la logica de SolicitudDao.php para asociar directamente el DNI ingresado durante el registro con la columna de cuenta de la tabla estudiantes. Ahora el pre-registro crea la solicitud de forma simultanea en usuarios (como pendiente) y en estudiantes (como pendiente con su respectivo DNI y carrera), eliminando la necesidad de generar cuentas ficticias.
  * **Capa de Modelos (DAO):** Se implemento el metodo existeCuenta en EstudianteDao.php para verificar si un DNI ya esta registrado en la base de datos por otro estudiante.
  * **Vistas y Controladores (`src/views/` y `src/controllers/`):** Creado el archivo UsuariosController.php para procesar las aprobaciones y rechazos de admision invocando a SolicitudDao, dejando el flujo completamente funcional para el maquetado del equipo.
  * **Vistas y Controladores (`src/controllers/`):** Se modifico EstudiantesController.php para validar la unicidad del DNI antes de insertar o actualizar registros de estudiantes, impidiendo excepciones de clave duplicada en PDO y devolviendo alertas amigables al usuario.
  * **Estructura de Trabajo:** Se crearon los archivos controladores y las plantillas de vistas (.tpl) en blanco correspondientes a los nuevos modulos para que el equipo pueda trabajar sin interferir entre si.
  * **Base de Datos (Migración):** Se realizo una migracion y limpieza en la base de datos: se creo la tabla facultades y se añadieron columnas de tipo_materia e id_facultad en materias e id_facultad en carreras para organizar el plan academico institucional (institucionales, de facultad y natas). Asimismo, se removieron todas las materias, secciones, matriculas y calificaciones previas para brindar un entorno limpio de pruebas, y se restablecio el estado de todos los alumnos en la tabla estudiantes a 'Admitido' y sus cuentas de usuario a 'activo'.
  * **Capa de Modelos (DAO):** Se modifico CarreraDao.php para incluir la sumatoria de materias activas especificas (natas, donde tipo_materia = 'carrera') en la consulta de obtenerCarreras, y se programo el metodo obtenerCarrerasParaRegistro el cual retorna unicamente aquellas carreras activas que cuentan con al menos una materia especifica asociada. Esto garantiza que una carrera sin clases especificas (aunque herede generales) se mantenga en estado "Pendiente de carga académica" y no este disponible para nuevos registros.
  * **Vistas y Controladores (`src/views/` y `src/controllers/`):** Se implementaron las vistas carreras.view.tpl y carreras_form.view.tpl con soporte para barra lateral dinamica y botones responsivos. Se programo CarrerasController.php para registrar, editar, activar e inactivar las carreras universitarias de la institucion, aplicando la regla de negocio para clasificar en estado "Pendiente de carga académica" a las carreras con 0 materias vinculadas e impedir que esten disponibles en el registro de estudiantes. Asimismo, se incorporo una validacion de unicidad para el nombre de la carrera en el controlador (invocando el metodo existeNombreCarrera de CarreraDao) para evitar colisiones e impedir excepciones de clave duplicada en la base de datos.
* **Estado final de los preparativos:** Completados con éxito. El entorno está listo y preparado para el trabajo distribuido del equipo.
